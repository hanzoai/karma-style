Karma — the native stack, verified

The Karma checkout flow is annotated with schema.org microdata and shadcn-style data-slot, emits standard GA4 ecommerce events through track.js to https://api.hanzo.ai/v1/analytics, checks out through the checkout.js widget, and records a privacy-masked session replay. This page fires each event live and shows the exact payload.

Packages (published to npm, vendored locally — no runtime CDN): track.js@0.2.19 → window.HanzoTrack, checkout.js@2.1.22 → window.HanzoCheckout. shop.js@3.0.31 / commerce.js@7.4.1 are the React/Next surface (shop.js wraps checkout.js); this vanilla storefront wires checkout.js directly.

3 · Checkout widget

checkout.js → checking…

On click, the storefront calls HanzoCheckout.create({apiKey}).createSession({lineItems}).redirectToCheckout() → hosted checkout at api.hanzo.ai/v1/checkout. Card fields live on that cross-origin page and never enter the Karma DOM, so replay cannot capture them by construction. Enabled by the CTO setting SPA_CHECKOUT_KEY (a publishable pk_… key).

4 · Session replay

Recorder → checking…

The recorder (hanzoai/rrweb blue/session-replay → insights.hanzo.ai) is not published yet, so replay ships as a wired, opt-in mount point. It lights up when the CTO sets SPA_REPLAY_SRC to the recorder URL — no code change. Privacy: opt-in consent, maskAllInputs, and hard-block on [data-slot="payment"] / .rr-block / data-private; email fields carry .rr-mask.
